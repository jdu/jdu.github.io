Don't Learn to Code, Learn to Use Better Tools

It comes in waves. Every few years, or some times just months, a wave of articles, news items and information all ultimately saying the same thing:

You should learn to code.

I've got nothing against learning to code, if you really want to, go for it.

There are better tools out there

There is a veritable cornucopia of tools out there for doing things like:

  • Wrangling data

  • Converting files between formats

  • Storing and retrieving information

  • Analysing data and information

Code is just a tool

Code is just one of a large number of tools that you can have in your arsenal, but it’s not the only one, and you can get away with a lot without every writing a single line of code

What Makes a Good Tool?

Simple interfaces that get out of your way

Tools should be simple to get started, but support a large amount of complexity if you really knuckle down and dig into it. If something is really hard and complex to get started with, it ceases being a tool. A lot of setup, configuration and a cludge of command-line arguments for instance will mean you’re less likely to reach for that tool when you need it, or you’ll have to do a lot of setup and maintain config and scripts around that tool in order to use it effectively at short notice.

Composable

An example of this is a lot of command-line tools whose output can be linked together using the "pipe" | operator. Things like cat some_file.txt | grep "some_string".

Good tools are "composable", that means they support linking together with other tools in some novel or easy to understand way, even back into itself, in order to perform complex operations or procedures.

Good Tools aren’t prescriptive about how you use them

A good tool will be designed in such a way that it is applicable to a wide variety of use-cases in it’s problem domain.

Focused on a specific problem or small problem space

Tools that try to do everything all at once, are really hard to manage, and become unstable really quickly. You have a lot of different requirements that at times can be at odds with each other. Tools that stick to doing one thing, or a small subset of related things are more likely ot be stable over time and will allow you a higher level of complexity in their small domain versus a tool that is schizophrenic about what it actually wants to be.
